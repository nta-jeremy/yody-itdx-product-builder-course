## A. Thiết kế

- **Chương trình:** A — Builder (Intern Product Builder)
- **Module / Level:** Module 4 (Tích hợp & Product Thinking) / **L4 Integrator**
- **Thời lượng:** 180 phút — **Hình thức:** live cohort + lab dài (code) + code review

- **Prerequisite:** đã có canvas Product Thinking của 1 feature (từ I4.1); biết cơ bản đọc/sửa code; có môi trường dev được cấp (sandbox Yody) và tài khoản Claude Code/API trong danh mục duyệt.

### Mục tiêu học tập (kết thúc buổi, học viên có thể…)

1. **Sử dụng** Claude Code / Claude API để **build 1 feature AI nhỏ chạy được** từ canvas Product Thinking của mình (input → AI → output có thể demo).
2. **Tích hợp** một AI component vào sản phẩm/prototype: gọi model, xử lý input/output, hiển thị kết quả cho người dùng.
3. **Thiết kế trust layer cơ bản** cho feature: kiểm tra/giới hạn input, xử lý khi AI trả lời sai/rỗng, fallback an toàn, và **ẩn PII trước khi gửi vào model**.
4. **Đánh giá** feature qua code review của peer/leader: đọc được điểm rủi ro của tích hợp AI và sửa ≥1 vấn đề.

> Mục tiêu 1–2 ánh xạ hành vi L4 (Can Do: tích hợp AI component vào sản phẩm). Mục tiêu 3 ánh xạ Can Do + Diligence (trust layer, data sensitivity) và là phôi cho guardrail ở I5.2. Mục tiêu 4 ánh xạ Will FIT (Good Relationship qua code review). Đây là buổi **tích lũy năng lực L4**.

### Trọng tâm 4D & chế độ AI

- **4D:** **Delegation (chính)** — giao đúng phần việc cho Claude Code, giữ phần kiểm soát; **Diligence (chính)** — trust layer, ẩn PII, không accept code/output blindly.
- **Chế độ AI:** Agency có kiểm soát — Claude Code sinh code, người định scope, review và chịu trách nhiệm kết quả.

## B. Triển khai — Agenda timeboxed (180′ · lý thuyết+demo 50′ ≈ 28% ≤ 30%)

| Thời gian | Hoạt động | Hình thức |
|---|---|---|
| 0–10′ | **Hook:** chiếu một feature AI tích hợp **không có trust layer** (gửi thẳng input khách kèm SĐT vào model, AI trả lời sai và app crash). Hỏi: "Hỏng ở đâu? Mấy chỗ?" | cả lớp |
| 10–30′ | **Lý thuyết:** kiến trúc một AI feature (input → tiền xử lý/redaction → gọi model → hậu xử lý/validate → output → fallback); trust layer là gì; vibe coding có kỷ luật với Claude Code | giảng |
| 30–60′ | **Demo theo bối cảnh Yody:** dùng Claude Code build feature "phân loại email khiếu nại" từ starter repo, có ẩn PII + fallback | trình diễn (live coding) |
| 60–150′ | **Lab thực hành (90′):** mỗi học viên build feature AI nhỏ từ canvas của mình + thêm trust layer | cá nhân (mentor đi vòng) |
| 150–170′ | **Code review cặp đôi:** rà theo bảng tiêu chí (trust layer, PII, fallback, secret), sửa ≥1 vấn đề | cặp đôi |
| 170–180′ | **Wrap-up + giao bài áp dụng** | cả lớp |

> Thực hành (lab 90′ + code review 20′ = **110′/180′ ≈ 61%**); cộng demo live-coding cùng làm theo → đạt mốc ~70%. Lý thuyết thuần giữ 28%.

### Nội dung cốt lõi (5 ý)

- **Một AI feature có 5 lớp, không chỉ "gọi model".** input → tiền xử lý (gồm redaction PII) → gọi model → hậu xử lý/validate output → output + fallback. Bỏ lớp nào cũng tạo rủi ro.
- **Trust layer là phần bao quanh model, không phải bản thân model.** Gồm: validate input, giới hạn phạm vi, kiểm tra output trước khi hiển thị, đường lui an toàn khi AI sai/rỗng/timeout.
- **Vibe coding có kỷ luật.** Claude Code sinh code nhanh, nhưng người build phải đọc-hiểu, review từng thay đổi, không merge mù. Tốc độ không thay thế trách nhiệm.
- **Secret & PII là ranh giới cứng.** Không hardcode API key (dùng biến môi trường); ẩn PII *trước khi* gửi vào model; log không chứa dữ liệu nhạy cảm.
- **Fallback định nghĩa trải nghiệm khi AI thất bại.** Feature tốt vẫn dùng được (hoặc báo lỗi tử tế) khi model trả lời sai, rỗng, hoặc service lỗi.

### Demo theo bối cảnh Yody

- **Bài toán:** feature "tự phân loại email khiếu nại của khách thành nhóm (đổi/trả, giao hàng, chất lượng, khác)".
- **Demo live coding với Claude Code:** từ starter repo, yêu cầu Claude Code (1) viết hàm redaction ẩn tên/SĐT/email trong nội dung trước khi gửi model, (2) gọi model phân loại với structured output, (3) validate nhãn trả về thuộc tập cho phép, (4) fallback "cần người xử lý" khi không chắc.
- **Chốt:** chỉ ra rõ đâu là model call và đâu là trust layer; nhấn rằng phần đáng giá của builder nằm ở trust layer, không phải ở dòng gọi model.

### Lab thực hành

- **Đề bài:** từ canvas Product Thinking của mình (I4.1), build **1 feature AI nhỏ chạy được** trong sandbox, kèm trust layer tối thiểu.
- **Dữ liệu:** dữ liệu giả lập + sandbox đã ẩn PII (không dữ liệu khách thật).
- **Tiêu chí ĐẠT:**
  - Feature **chạy được** end-to-end: nhập input → ra output demo được.
  - Có **trust layer**: ≥1 cơ chế validate input/output + ≥1 fallback khi AI sai/rỗng.
  - Có **redaction PII** trước khi gửi model + **không hardcode secret** (dùng biến môi trường).

### ⚠️ An toàn dữ liệu (bắt buộc)

- **Ẩn PII trước khi gửi vào model** là yêu cầu cứng của lab — không phải tùy chọn. Redaction phải xử lý tổ hợp định danh (tên + SĐT + địa chỉ + email + CCCD).
- **Tuyệt đối không hardcode API key/secret** vào repo; dùng biến môi trường; không commit file `.env`.
- Log/telemetry **không** được chứa nội dung PII. Theo **Luật 91/2025/QH15** (hiệu lực 01/01/2026) và **NĐ 356/2025/NĐ-CP**, xử lý PII trái phép có thể bị chế tài.
- Mọi dữ liệu lab là **giả lập** hoặc sandbox đã ẩn PII; cấm kéo dữ liệu production thật vào buổi.

## Ghi chú cho người dạy (facilitator notes)

- **Điểm học viên hay vấp:** (1) "chạy được là xong", bỏ trust layer → dùng Hook để cho thấy feature không trust layer vỡ thế nào. (2) merge code Claude Code sinh ra mà không đọc → bắt giải thích từng đoạn khi review. (3) quên redaction, gửi thẳng input có PII → đây là lỗi chặn ĐẠT, không thương lượng.
- **Câu hỏi thường gặp:** "Feature nhỏ tới đâu là đủ?" → một input → một AI call có ý nghĩa → một output demo được, kèm trust layer; không cần UI đẹp. "Dùng Claude Code hay API trực tiếp?" → tùy độ quen; người mới nên dùng Claude Code để đi nhanh, vẫn phải đọc code.
- **Phương án dự phòng:** API rate limit/lỗi → dùng response mẫu cache sẵn để tập phần trust layer & fallback; sandbox lỗi → ghép đôi build chung trên một máy.
- **Lưu ý gate:** đây chưa phải buổi gate; nhấn feature + trust layer hôm nay là **vật liệu** cho vòng iterate ở I4.3.
- **Nối rubric:** tín hiệu mạnh cho Can Do (workflow design + tích hợp AI) và Diligence (trust layer, PII); leader ghi nhận vào đánh giá giữa kỳ.
