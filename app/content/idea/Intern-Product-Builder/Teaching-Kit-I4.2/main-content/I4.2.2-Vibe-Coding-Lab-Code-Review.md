### Bài toán: Feature phân loại email khiếu nại của Yody

**Vấn đề thực tế:** Team CSKH của Yody nhận hàng trăm email khiếu nại mỗi ngày. Việc đọc và phân loại từng email thủ công mất nhiều thời gian và dễ nhầm lẫn. Một AI feature có thể tự động phân loại email vào đúng nhóm để routing đến đúng bộ phận xử lý.

**Kết quả mong đợi:** Nhập nội dung email vào → nhận về một trong các nhãn: `doi_tra` (đổi/trả), `giao_hang` (giao hàng), `chat_luong` (chất lượng), `khac` (khác), hoặc `can_nguoi_xu_ly` (fallback khi có lỗi hoặc độ tin cậy thấp).

### Kiến trúc pipeline 5 lớp sẽ build:

```
[Lớp 1] Input email (nội dung text gốc)
    ↓
[Lớp 2] Redact PII (ẩn SĐT, email, CCCD)
    ↓
[Lớp 3] Gọi model (structured output JSON)
    ↓
[Lớp 4] Validate nhãn (có trong tập cho phép? độ tin cậy đủ?)
    ↓
[Lớp 5] Trả nhãn / Fallback "cần người xử lý"
```

### Bước 2 — Build lớp Redaction (Lớp 2)

Đây là lớp đầu tiên và quan trọng nhất cần build. Nhớ: PII phải bị ẩn **trước khi** làm bất cứ điều gì khác.

```python
# phan_loai_khieu_nai.py

import re
import os
import json
import anthropic
from dotenv import load_dotenv

# Load biến môi trường từ file .env
load_dotenv()

# ============================================================
# LỚP 2: TIỀN XỬ LÝ + REDACTION PII
# ============================================================

def redact_pii(text: str) -> str:
    """
    Ẩn thông tin định danh cá nhân (PII) trong text trước khi gửi model.
    
    Xử lý:
    - Số điện thoại VN (10 số, bắt đầu 0) → [SĐT]
    - Địa chỉ email → [EMAIL]
    - Số CCCD/CMND (12 chữ số) → [CCCD]
    
    Giới hạn: Chưa ẩn tên người và địa chỉ — cần NER cho bản production.
    """
    # Ẩn số điện thoại VN: 0xxxxxxxxx (10 số)
    text = re.sub(r"\b0\d{9}\b", "[SĐT]", text)
    
    # Ẩn địa chỉ email
    text = re.sub(r"[\w.+\-]+@[\w\-]+\.[\w.\-]+", "[EMAIL]", text)
    
    # Ẩn số CCCD/CMND (12 chữ số liên tiếp)
    text = re.sub(r"\b\d{12}\b", "[CCCD]", text)
    
    return text


def validate_input(text: str) -> bool:
    """
    Kiểm tra input có hợp lệ không.
    
    Returns:
        True nếu hợp lệ, False nếu không
    """
    if not text or not text.strip():
        return False  # Không được rỗng
    if len(text.strip()) < 10:
        return False  # Quá ngắn
    if len(text) > 2000:
        return False  # Quá dài
    return True
```

### Bước 3 — Build lớp Gọi Model (Lớp 3)

Chỉ build lớp này sau khi lớp 2 đã hoạt động đúng.

```python
# Thêm vào phan_loai_khieu_nai.py

# ============================================================
# LỚP 3: GỌI MODEL VỚI STRUCTURED OUTPUT
# ============================================================

# Tập nhãn cho phép — là nguồn của sự thật (source of truth)
# Mọi validation đều so với tập này
TAP_NHAN_HOP_LE = {"doi_tra", "giao_hang", "chat_luong", "khac"}

# System prompt — giới hạn phạm vi của model
SYSTEM_PROMPT = """
Bạn là hệ thống phân loại khiếu nại khách hàng của Yody — thương hiệu thời trang Việt Nam.

NHIỆM VỤ DUY NHẤT của bạn: Đọc nội dung khiếu nại và phân loại vào đúng một trong bốn nhóm:
- doi_tra: Khách muốn đổi hoặc trả sản phẩm, hoàn tiền
- giao_hang: Vấn đề về giao hàng, vận chuyển, thời gian nhận hàng
- chat_luong: Phàn nàn về chất lượng sản phẩm (vải, đường chỉ, màu sắc, kích thước)
- khac: Các nội dung khác không thuộc ba nhóm trên

ĐỊNH DẠNG TRẢ VỀ (bắt buộc, chỉ JSON, không thêm gì khác):
{"nhan": "<một trong bốn nhãn trên>", "do_tin_cay": <số thực từ 0.0 đến 1.0>}

VÍ DỤ:
Input: "Tôi đặt áo khoác [SĐT] nhưng giao sai size, muốn đổi"
Output: {"nhan": "doi_tra", "do_tin_cay": 0.95}

KHÔNG được: trả lời, giải thích, tư vấn, hay làm bất cứ điều gì khác ngoài phân loại.
"""


def goi_model(noi_dung_da_redact: str) -> dict | None:
    """
    Gọi Claude model để phân loại nội dung đã redact.
    
    Args:
        noi_dung_da_redact: Nội dung khiếu nại đã được ẩn PII
    
    Returns:
        dict với 'nhan' và 'do_tin_cay', hoặc None nếu thất bại
    """
    # Đọc API key từ biến môi trường — KHÔNG hardcode
    api_key = os.environ.get("CLAUDE_API_KEY")
    model_name = os.environ.get("MODEL_NAME", "claude-opus-4-5")
    
    if not api_key:
        print("[ERROR] CLAUDE_API_KEY chưa được cấu hình")
        return None
    
    try:
        client = anthropic.Anthropic(api_key=api_key)
        
        response = client.messages.create(
            model=model_name,
            max_tokens=100,  # Chỉ cần output JSON ngắn
            system=SYSTEM_PROMPT,
            messages=[
                {"role": "user", "content": noi_dung_da_redact}
            ]
        )
        
        # Lấy text từ response
        raw_text = response.content[0].text.strip()
        
        # Parse JSON
        return json.loads(raw_text)
        
    except anthropic.APITimeoutError:
        print("[WARN] API timeout")
        return None
    except anthropic.APIError as e:
        print(f"[WARN] API error: {e}")
        return None
    except json.JSONDecodeError as e:
        print(f"[WARN] Response không phải JSON hợp lệ: {e}")
        return None
```

### Bước 4 — Build lớp Validate + Fallback (Lớp 4 + 5)

Đây là lớp ghép lại tất cả và xử lý mọi tình huống.

```python
# Thêm vào phan_loai_khieu_nai.py

# ============================================================
# LỚP 4: VALIDATE OUTPUT
# ============================================================

NGUONG_TIN_CAY = 0.6  # Model phải ít nhất 60% chắc chắn

def ket_qua_hop_le(r) -> bool:
    """
    Kiểm tra output từ model có hợp lệ để dùng không.
    
    Điều kiện hợp lệ:
    1. Phải là dict (JSON đã parse thành công)
    2. Phải có trường 'nhan'
    3. Nhãn phải nằm trong tập cho phép
    4. Độ tin cậy phải trên ngưỡng tối thiểu
    """
    if not isinstance(r, dict):
        return False
    
    if "nhan" not in r:
        return False
    
    if r["nhan"] not in TAP_NHAN_HOP_LE:
        return False
    
    if r.get("do_tin_cay", 0) < NGUONG_TIN_CAY:
        return False
    
    return True


# ============================================================
# LỚP 5: OUTPUT + FALLBACK — HÀM CHÍNH
# ============================================================

def phan_loai_khieu_nai(noi_dung_email: str) -> str:
    """
    Hàm chính: phân loại email khiếu nại qua toàn bộ pipeline 5 lớp.
    
    Quy trình:
    1. Validate input (Lớp 1+2 đầu)
    2. Redact PII (Lớp 2)
    3. Gọi model (Lớp 3)
    4. Validate output (Lớp 4)
    5. Trả nhãn hoặc fallback (Lớp 5)
    
    Returns:
        Một trong: 'doi_tra', 'giao_hang', 'chat_luong', 'khac', 'can_nguoi_xu_ly'
    """
    # --- BƯỚC 1: Validate input (Lớp 1+2)
    if not validate_input(noi_dung_email):
        print("[INFO] Input không hợp lệ → fallback")
        return "can_nguoi_xu_ly"
    
    # --- BƯỚC 2: Redact PII (Lớp 2)
    # QUAN TRỌNG: bước này PHẢI chạy trước khi gọi model
    noi_dung_sach = redact_pii(noi_dung_email)
    
    # Log để debug — chú ý: log nội dung ĐÃ redact, không log nội dung gốc
    print(f"[DEBUG] Nội dung sau redact: {noi_dung_sach[:100]}...")
    
    # --- BƯỚC 3: Gọi model (Lớp 3)
    ket_qua = goi_model(noi_dung_sach)
    
    # --- BƯỚC 4 + 5: Validate và Fallback
    if ket_qua is None:
        # Model không phản hồi hoặc có lỗi
        print("[INFO] Không nhận được kết quả từ model → fallback")
        return "can_nguoi_xu_ly"
    
    if not ket_qua_hop_le(ket_qua):
        # Kết quả không hợp lệ hoặc không đủ tin cậy
        print(f"[INFO] Kết quả không đủ điều kiện (kết quả: {ket_qua}) → fallback")
        return "can_nguoi_xu_ly"
    
    # Kết quả tốt — trả nhãn
    nhan = ket_qua["nhan"]
    do_tin_cay = ket_qua["do_tin_cay"]
    print(f"[INFO] Phân loại thành công: {nhan} (độ tin cậy: {do_tin_cay:.0%})")
    return nhan
```

### Đề bài

Lấy **canvas Product Thinking** từ buổi I4.1 của bạn. Chọn **một feature AI** từ canvas đó và build nó thành một script Python chạy được, kèm trust layer tối thiểu.

**Yêu cầu kỹ thuật tối thiểu:**
- **Một luồng rõ ràng:** một input → một AI call có nghĩa → một output demo được
- **Redaction PII:** nếu input có thể chứa thông tin cá nhân, phải có bước redact trước khi gửi model
- **Validate output:** kiểm tra output từ model trước khi dùng
- **Fallback:** ít nhất một đường lui khi AI sai/rỗng/timeout
- **Không hardcode secret:** API key phải đọc từ biến môi trường
- **Có thể giải thích từng hàm trong code:** khi mentor/reviewer hỏi, bạn phải giải thích được từng đoạn đang làm gì — không merge mù

**Không cần:**
- UI đẹp — terminal output is enough
- Feature phức tạp — một input → một output is enough
- 100% accuracy — mục tiêu là kiến trúc đúng, không phải kết quả hoàn hảo

### Checklist Trust Layer — Tự rà khi build

In ra và check từng mục khi build xong:

- [ ] **Validate input:** Input được kiểm tra trước khi xử lý (độ dài, định dạng, không rỗng)
- [ ] **Redaction PII:** Tất cả PII (SĐT, email, CCCD) được ẩn **trước khi** gửi model
- [ ] **Structured output:** Model được yêu cầu trả định dạng cố định (JSON) — không phải text tự do
- [ ] **Validate output:** Output từ model được kiểm tra định dạng và giá trị trước khi dùng
- [ ] **Fallback hoạt động:** Test tắt mạng hoặc truyền input mơ hồ — hệ thống không crash
- [ ] **Secret an sau:** API key được đọc từ biến môi trường, không hardcode trong code
- [ ] **Log sạch:** Kiểm tra log output — không chứa PII gốc (chỉ chứa PII đã redact)

### Tiêu chí ĐẠT

Feature của bạn đạt tiêu chí nếu đáp ứng **cả ba điều kiện** sau:

**1. Chạy được end-to-end**
- Chạy script và truyền input → nhận được output có nghĩa, không crash
- Có ít nhất 2-3 test case chạy được

**2. Có trust layer thật**
- Ít nhất một cơ chế validate input (không phải comment — phải là code chạy được)
- Ít nhất một cơ chế validate output (kiểm tra nhãn, định dạng, hoặc ngưỡng tin cậy)
- Ít nhất một fallback có ý nghĩa khi AI sai/rỗng (không phải `return None`)

**3. Không vi phạm ranh giới cứng**
- Có hàm redact PII và nó được gọi trước khi truyền vào model (nếu input có thể chứa PII)
- API key không hardcode trong code — đọc từ biến môi trường
- `.env` không được commit (có trong `.gitignore`)

## 10. Code Review cặp đôi — Đọc code người khác như thế nào

### Tại sao code review quan trọng?

Code review không phải là tìm lỗi để "bắt bẻ" nhau — đó là một cách học và một lớp bảo vệ thứ hai.

Khi bạn build một feature, bạn biết nó hoạt động như thế nào — bạn đã test các ca bạn nghĩ đến. Người review sẽ nhìn thấy những ca bạn chưa nghĩ đến, những assumption bạn đang làm ngầm, và những edge case bạn đã bỏ qua.

Đặc biệt với AI feature, code review còn giúp kiểm tra trust layer — thứ rất dễ bị bỏ qua khi viết.

### Bốn điểm cần rà khi review AI feature

**1. Trust layer có đủ không?**

Hỏi người viết:
- Input được validate chưa? Validate gì? Ở đâu trong code?
- Output được kiểm tra trước khi dùng chưa? Kiểm tra cụ thể cái gì?
- Nếu model trả về định dạng khác mong đợi, code xử lý thế nào?

Tìm kiếm trong code: Có `if` statement nào kiểm tra output không? Hay code dùng trực tiếp `result["key"]` không có bước kiểm tra trước?

---

**2. PII có được ẩn đúng không?**

Hỏi người viết:
- Input của feature có thể chứa PII không?
- Nếu có, hàm redact được gọi ở đâu?
- Hàm redact có được gọi **trước** khi truyền vào model không?

Tìm kiếm trong code: Tìm hàm gọi model. Nhìn vào input được truyền vào đó. Nó có đi qua bước redact trước không? Hay nó là raw input từ người dùng?

Thử nghiệm: Truyền input có SĐT thật (giả lập). Kiểm tra log — SĐT có bị ẩn không?

---

**3. Fallback có thật không?**

Hỏi người viết:
- Nếu model timeout, code xử lý thế nào?
- Nếu model trả về format khác mong đợi, code xử lý thế nào?
- "Fallback" trong code có thực sự xử lý được hay chỉ là `return None` hoặc `pass`?

Thử nghiệm thực tế:
- Ngắt mạng và chạy — hệ thống crash hay trả fallback?
- Truyền input mơ hồ hoặc ngắn vô nghĩa — hệ thống xử lý thế nào?

---

**4. Secret có an toàn không?**

Kiểm tra trực tiếp trong code:
- Có chuỗi nào trông như API key không? (bắt đầu bằng `sk-`, `Bearer `, dài 40+ ký tự)
- Key được đọc từ `os.environ` hay từ biến hardcoded?
- File `.env` có trong `.gitignore` không?

```bash
# Lệnh kiểm tra nhanh
grep -rn "sk-" .                    # Tìm chuỗi trông như API key
grep -rn "api_key\s*=" .            # Tìm assignment api_key
cat .gitignore | grep ".env"        # Kiểm tra .env có trong gitignore không
```

### Quy trình code review cặp đôi (20 phút)

**Phân chia vai:** Người A review code của Người B, và ngược lại.

**Bước 1 (5 phút):** Người viết chạy demo feature của mình — show end-to-end và ít nhất một ca fallback.

**Bước 2 (10 phút):** Người review đọc code và hỏi về bốn điểm: trust layer, redaction PII, fallback, secret. Người viết giải thích từng phần.

> **Quy tắc quan trọng:** Nếu người viết không giải thích được một đoạn code, đó là dấu hiệu của "vibe coding không kỷ luật" — merge code mà chưa đọc-hiểu. Đây là thứ cần sửa.

**Bước 3 (5 phút):** Thống nhất ít nhất một vấn đề cần sửa. Người viết sửa ngay trong buổi.

### Sau code review: sửa và ghi chú

Mỗi người cần:
1. Sửa ít nhất một vấn đề được chỉ ra
2. Thêm comment giải thích vào những đoạn code phức tạp
3. Ghi lại: trust layer của feature có những cơ chế gì (để mang sang I4.3)

## 11. Tóm tắt & bước tiếp theo

### Năm điều cốt lõi từ buổi hôm nay

**1. AI feature có 5 lớp, không chỉ là "gọi model"**

Đầu vào → Tiền xử lý & Redaction → Gọi model → Hậu xử lý & Validate → Đầu ra & Fallback. Bốn trong năm lớp là trust layer. Chỉ một lớp là gọi AI.

**2. Trust layer là phần đáng giá, không phải dòng gọi model**

Ai cũng gọi model được sau 5 phút học API. Điều phân biệt builder level 4 là biết bảo vệ hệ thống, dữ liệu, và người dùng khỏi những lúc AI thất bại.

**3. Vibe coding phải có kỷ luật**

Claude Code sinh code nhanh — đó là sức mạnh. Nhưng bạn phải đọc-hiểu từng thay đổi trước khi giữ. Tốc độ không thay thế trách nhiệm.

**4. Secret và PII là ranh giới cứng**

Không hardcode API key. Luôn redact PII trước khi gửi model. Log không chứa PII. Ba quy tắc này không có ngoại lệ trong bất kỳ feature nào ở Yody.

**5. Fallback tạo nên trải nghiệm khi AI sai**

AI sẽ có lúc sai — đây là thực tế, không phải bug. Feature tốt là feature có đường lui duyên dáng khi điều đó xảy ra.

### Bài tập về nhà (chuẩn bị cho I4.3)

1. **Hoàn thiện feature** từ lab hôm nay — đảm bảo toàn bộ checklist trust layer được check
2. **Gắn feature vào sáng kiến thật** của bạn tại Yody (đã ẩn PII, chạy được với dữ liệu giả lập)
3. **Ghi chú trust layer** — mô tả feature của bạn có cơ chế nào ở mỗi lớp
4. **Cho mentor xem** trước buổi I4.3

### Buổi tiếp theo: I4.3 — Tích hợp Sáng kiến & Iterate (Gate L4→L5)

Buổi I4.3 là buổi **gate** — feature bạn build hôm nay sẽ được đưa vào một sáng kiến thật có stakeholder ngoài team. Bạn sẽ thu feedback và iterate ít nhất 2 vòng để tạo bằng chứng cho Gate L4→L5.

Feature hôm nay là vật liệu cho buổi đó — hãy build cho tốt.

## 12. Phụ lục: Dữ liệu mẫu để thực hành

> ⚠️ **Lưu ý:** Tất cả tên, số điện thoại, email, CCCD dưới đây đều là **dữ liệu giả lập hoàn toàn** — chỉ dùng để test. Tuyệt đối không thay thế bằng dữ liệu thật của khách hàng Yody trong bất kỳ buổi lab nào.

### Sáu email khiếu nại mẫu

**Email #1 — Đổi/trả + có PII (test redaction)**
```
Tôi tên Trần Văn Demo, SĐT 0901234567, đặt áo khoác nhưng giao sai size, muốn đổi.
```
*Kỳ vọng:* Nhãn `doi_tra`  
*Test gì:* SĐT `0901234567` phải thành `[SĐT]` trong log

---

**Email #2 — Giao hàng + có email (test redaction)**
```
Đơn của tôi 5 ngày chưa giao, ai xử lý giúp? Email demo@example.com
```
*Kỳ vọng:* Nhãn `giao_hang`  
*Test gì:* `demo@example.com` phải thành `[EMAIL]` trong log

---

**Email #3 — Chất lượng (không có PII)**
```
Vải áo bị bung chỉ sau một lần giặt, chất lượng kém.
```
*Kỳ vọng:* Nhãn `chat_luong`  
*Test gì:* Không có PII để redact — nhưng redact function phải chạy mà không lỗi

---

**Email #4 — Khác + có CCCD (test redaction)**
```
Cho hỏi CCCD 012345678901 của tôi đã được xác minh chưa? Tôi cần làm thủ tục đổi trả.
```
*Kỳ vọng:* Nhãn `khac`  
*Test gì:* `012345678901` phải thành `[CCCD]` trong log

---

**Email #5 — Mơ hồ (test fallback)**
```
Tôi muốn vừa đổi hàng vừa hỏi sao giao chậm.
```
*Kỳ vọng:* Có thể là `doi_tra`, `giao_hang`, hoặc `can_nguoi_xu_ly` — cả ba đều chấp nhận được  
*Test gì:* Model có nhận ra đây là case mơ hồ và phản ánh qua độ tin cậy thấp không?

---

**Email #6 — Không phàn nàn**
```
Sản phẩm tốt, không có gì phàn nàn, cảm ơn shop.
```
*Kỳ vọng:* Nhãn `khac`  
*Test gì:* Model có phân loại đúng nội dung tích cực vào `khac` không?
