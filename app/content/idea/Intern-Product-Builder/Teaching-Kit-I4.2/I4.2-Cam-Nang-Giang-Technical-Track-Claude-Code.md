## 0. Định vị buổi học & cách dùng cẩm nang

**Cách dùng tài liệu:** Phần 1 talk-track. Phần 2 kịch bản demo live-coding với Claude Code — có code mẫu (Python) để người dạy đọc và làm theo. Phần 3 lab + checklist trust layer + tiêu chí ĐẠT. Phần 4 dữ liệu giả lập in phát. Phần 5 code review + đóng gói. Phần 6 FAQ. Phần 7 image prompts.

> ⚠️ **Lưu ý kỹ thuật:** code mẫu là *pseudocode minh hoạ kiến trúc*, không phải code production. Mục tiêu dạy là **5 lớp của một AI feature** và **trust layer**, không phải cú pháp một SDK cụ thể. Trước buổi kiểm lại tên model/SDK hiện hành; dạy nguyên lý, không bắt thuộc API.

**Buổi này ở đâu trong lộ trình:** sau I4.1 (đã có canvas Product Thinking). I4.2 là buổi **kỹ thuật**: học viên dùng Claude Code/Claude API build feature AI nhỏ *chạy được* từ canvas của mình, kèm **trust layer** (validate input/output, fallback, ẩn PII, không hardcode secret). Đây là buổi tích lũy; feature + trust layer hôm nay là vật liệu cho vòng iterate ở I4.3 (gate L4→L5).

### Khác biệt cốt lõi (nhắc lại ≥3 lần trong buổi)

- **Người mới hỏi:** "Feature chạy được chưa?"
- **Builder có kỷ luật hỏi:** "Khi AI trả lời sai/rỗng/timeout thì sao? PII đã ẩn trước khi vào model chưa? Secret có lộ trong repo không?"

> Thông điệp neo cả buổi: **phần đáng giá của builder nằm ở trust layer, không phải ở dòng gọi model.** "Chạy được" chưa phải "dùng được".

### Bảng xương sống: một AI feature có 5 lớp

| Lớp | Làm gì | Bỏ lớp này → rủi ro |
|---|---|---|
| **1. Input** | Nhận đầu vào người dùng | — |
| **2. Tiền xử lý + redaction PII** | Validate input, **ẩn PII trước khi vào model** | Lộ PII vào model/log |
| **3. Gọi model** | Gọi AI với structured output | — |
| **4. Hậu xử lý + validate** | Kiểm output đúng định dạng/phạm vi | Output sai/độc lọt tới người dùng |
| **5. Output + fallback** | Trả kết quả; đường lui khi AI sai/rỗng/timeout | App crash hoặc trả rác |

> **Nguyên tắc buổi:** 30/70 (lý thuyết ≤28%) · 4D trọng tâm = **Delegation (chính)** + **Diligence (chính)** · chế độ **Agency có kiểm soát** (Claude Code sinh code, người định scope, review và chịu trách nhiệm).

## 2. Kịch bản demo live-coding (Claude Code + trust layer)

**Bài toán:** feature "tự phân loại email khiếu nại của khách thành nhóm (đổi/trả, giao hàng, chất lượng, khác)". Dữ liệu giả lập ở Phần 4.

**Chuẩn bị:** mở Claude Code trong starter repo; key cấp qua biến môi trường (KHÔNG trong repo); có file `khieu-nai-mau.md` (Phần 4).

**Bước 1 — Lớp redaction (ẩn PII trước khi gửi model):**
> *Yêu cầu Claude Code:* "Viết hàm `redact(text)` ẩn tên, SĐT, email, địa chỉ trong nội dung khiếu nại trước khi gửi model. Trả về text đã ẩn + map placeholder."

```python
import re

def redact(text: str) -> str:
    text = re.sub(r"\b0\d{9}\b", "[SĐT]", text)            # số điện thoại VN
    text = re.sub(r"[\w.+-]+@[\w-]+\.[\w.-]+", "[EMAIL]", text)
    text = re.sub(r"\b\d{12}\b", "[CCCD]", text)           # CCCD 12 số
    # tên/địa chỉ: dùng danh sách/NER ở bản nâng cấp; demo minh hoạ regex cơ bản
    return text
```
> *Chốt:* "Đây là lớp 2. Mọi nội dung khách đi qua đây *trước khi* tới model. Bản production cần NER cho tên/địa chỉ; demo dùng regex để minh hoạ nguyên lý."

**Bước 2 — Gọi model với structured output (lớp 3):**
> *Yêu cầu Claude Code:* "Gọi model phân loại `redact(noi_dung)` vào đúng một nhãn trong tập `['doi_tra','giao_hang','chat_luong','khac']`. Bắt model trả JSON `{nhãn, độ_tin_cậy}`."

```python
ALLOWED = {"doi_tra", "giao_hang", "chat_luong", "khac"}

def classify(noi_dung: str) -> dict:
    safe = redact(noi_dung)
    resp = call_model(prompt=PROMPT_PHAN_LOAI, input=safe)  # trả JSON {nhan, do_tin_cay}
    return resp
```

**Bước 3 — Validate output + fallback (lớp 4–5):**
> *Yêu cầu Claude Code:* "Kiểm nhãn trả về có thuộc ALLOWED không và độ tin cậy ≥ ngưỡng; nếu không, fallback 'cần người xử lý'."

```python
def phan_loai_an_toan(noi_dung: str) -> str:
    try:
        r = classify(noi_dung)
        if r.get("nhan") in ALLOWED and r.get("do_tin_cay", 0) >= 0.6:
            return r["nhan"]
        return "can_nguoi_xu_ly"        # fallback: không chắc → chuyển người
    except (TimeoutError, ValueError):
        return "can_nguoi_xu_ly"        # fallback: lỗi/timeout
```

**Bước 4 — Chạy thử + thử ca xấu:**
> Chạy với email thường → ra nhãn đúng. Chạy với email có SĐT → kiểm log: SĐT đã thành `[SĐT]`. Chạy với email mơ hồ → ra `can_nguoi_xu_ly` (fallback hoạt động).

**Rút ra:** *"Đếm lại: bốn trên năm lớp là trust layer — redaction, validate, fallback. Dòng gọi model chỉ một dòng. Phần đáng giá của builder nằm ở trust layer. Và để ý: tôi *đọc* từng đoạn Claude Code sinh ra trước khi giữ — vibe coding có kỷ luật."*

> **Dự phòng demo:** API rate limit/lỗi → dùng response mẫu cache sẵn để tập phần trust layer & fallback; sandbox lỗi → ghép đôi build chung trên một máy.

## 3. Lab build (90′) + checklist trust layer + tiêu chí ĐẠT

### Đề bài
Từ canvas Product Thinking (I4.1), build **1 feature AI nhỏ chạy được** trong sandbox, kèm **trust layer tối thiểu**.

### Checklist trust layer (in phát — tự rà khi build)
- [ ] **Validate input:** kiểm đầu vào đúng dạng/độ dài/phạm vi trước khi xử lý.
- [ ] **Redaction PII:** ẩn tên + SĐT + email + địa chỉ + CCCD **trước khi gửi model**.
- [ ] **Structured output:** model trả định dạng kiểm được (JSON/nhãn trong tập cho phép).
- [ ] **Validate output:** kiểm output đúng định dạng + thuộc phạm vi cho phép.
- [ ] **Fallback:** đường lui khi AI sai/rỗng/timeout (không crash, không gán bừa).
- [ ] **Secret:** API key qua biến môi trường; không commit `.env`.
- [ ] **Log sạch:** log không chứa PII.

### Tiêu chí ĐẠT
- Feature **chạy được end-to-end**: nhập input → ra output demo được.
- Có **trust layer**: ≥1 cơ chế validate input/output + ≥1 fallback khi AI sai/rỗng.
- Có **redaction PII** trước khi gửi model + **không hardcode secret** (dùng biến môi trường).

### Code review cặp đôi (20′)
Rà theo **4 điểm**: trust layer, redaction PII, fallback, secret handling. Mỗi người **sửa ≥1 vấn đề** được chỉ ra. Người review phải *đọc-hiểu* code, hỏi tác giả giải thích từng đoạn (chống merge mù).

### ⚠️ An toàn dữ liệu (bắt buộc)
Ẩn PII trước khi gửi model là **yêu cầu cứng** (xử lý tổ hợp định danh). **Tuyệt đối không hardcode API key/secret**, không commit `.env`. Log không chứa PII. Dữ liệu lab **giả lập**/sandbox đã ẩn PII — cấm kéo production thật vào buổi.

## 4. Dữ liệu mẫu (giả lập, KHÔNG PII) — in phát

> ⚠️ **Bịa hoàn toàn**. Tên/SĐT/email bên dưới là *giả*, dùng để test redaction — cấm thay bằng dữ liệu khách thật.

### 4.1 Sáu email khiếu nại giả lập (để test phân loại + redaction)
> 1. "Tôi tên Trần Văn Demo, SĐT 0901234567, đặt áo khoác nhưng giao sai size, muốn đổi." *(đổi/trả + có PII để test redaction)*
> 2. "Đơn của tôi 5 ngày chưa giao, ai xử lý giúp? Email demo@example.com" *(giao hàng + email)*
> 3. "Vải áo bị bung chỉ sau một lần giặt, chất lượng kém." *(chất lượng)*
> 4. "Cho hỏi cửa hàng gần CCCD 012345678901 của tôi có mở cửa chủ nhật không?" *(khác + CCCD để test redaction)*
> 5. "Tôi muốn vừa đổi hàng vừa hỏi sao giao chậm." *(mơ hồ → test fallback 'cần người')*
> 6. "Sản phẩm tốt, không có gì phàn nàn, cảm ơn shop." *(khác)*

### 4.2 Tập nhãn cho phép
> `doi_tra` · `giao_hang` · `chat_luong` · `khac` · *(fallback)* `can_nguoi_xu_ly`

### 4.3 Mẫu `.env` (KHÔNG commit)
```
CLAUDE_API_KEY=...    # cấp trong sandbox, KHÔNG hardcode vào code, KHÔNG commit file này
MODEL_NAME=...        # kiểm tên model hiện hành trước buổi
```
> *Mẹo:* email #1 và #4 là ca *bắt buộc* kiểm log sau khi chạy — SĐT phải thành `[SĐT]`, CCCD thành `[CCCD]`. Email #5 là ca kiểm fallback.

## 6. Ghi chú cho người dạy & FAQ

### Học viên hay vấp
- **"Chạy được là xong", bỏ trust layer** → dùng Hook để cho thấy feature không trust layer vỡ thế nào.
- **Merge code Claude Code sinh ra mà không đọc** → bắt giải thích từng đoạn khi review; vibe coding có kỷ luật.
- **Quên redaction, gửi thẳng input có PII** → lỗi *chặn ĐẠT*, không thương lượng.

### FAQ
- *"Feature nhỏ tới đâu là đủ?"* → Một input → một AI call có nghĩa → một output demo được, kèm trust layer; không cần UI đẹp.
- *"Dùng Claude Code hay API trực tiếp?"* → Tùy độ quen; người mới nên dùng Claude Code để đi nhanh, vẫn phải đọc code.
- *"Redaction regex có đủ không?"* → Đủ để minh hoạ nguyên lý ở lab; production cần NER cho tên/địa chỉ — ghi rõ là giới hạn để nâng cấp.

### Phương án dự phòng
- **API rate limit/lỗi** → dùng response mẫu cache sẵn để tập trust layer & fallback.
- **Sandbox lỗi** → ghép đôi build chung trên một máy.
